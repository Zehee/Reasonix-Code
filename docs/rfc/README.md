# 让 Reasonix 射出穿越时间的箭：跨会话主题追溯方案

> **一句话定义**: 系统自动索引所有历史 session，让 Agent 可以跨时间线搜索和追溯"同一个主题的完整发展史"。  
> 提炼零 LLM（低成本+可复现），挂载用 LLM（归类即理解）。

> **状态**：草案 v1 · 2026-06-25
> **性质**：Reasonix 社区设计文档

---

## 一、一个基本事实

当前市面上agents的两大机制决定了对话历史的**碎片化命运**：

**机制一：上下文压缩（Compaction）**

每次会话中，当 Agent 的上下文窗口接近上限时，Reasonix 会自动压缩早期内容。压缩后最早的对话轮丢失，只保留最近的上下文。

真实场景：你花了**一周乃至数月**做一个 Auth 模块——从方案讨论到最终上线，经历了设计、架构选型、编码实现、测试、迭代、修复。沿途的决策过程和理由——JWT 方案选型、为什么选了 httpOnly cookie 而不是 localStorage、Safari 兼容的折衷——在压缩发生时就已经在当前会话中被割裂。如果它们跨了多个 session（很可能的事），每次 session 边界都是一次新的割裂。

> 压缩只影响当前会话。完整的对话一直保存在 `archive/` 下的 JSONL 文件中——但 Agent 不会主动去读，因为它不知道"该查什么"。

**机制二：会话独立**

每次新会话启动时，上下文从零开始。Agent 能直接看到的只有系统提示词、项目文件、AGENTS.md，以及已保存的 `memory/` 条目。**所有这些只保存手动标记的结论，不保存决策过程**——讨论中的否决方案、中间推理、让步理由，都留在了上一个 session 的 JSONL 里。

**两个机制叠加的效果**：一次跨 session 的重大架构决策，最终散落在 N 个 JSONL 文件的碎片中——其中某些碎片还被 compaction 截断了——Agent 没有能力把它们打捞出来、串成完整的时间线。

> 这里说的"一次会话"就是一次 `reasonix-code run` 或一次连续对话。会话里的一次完整交互——用户提问、Agent 思考、调用工具、返回结果——称为一个 **turn**。

这就是本设计要解决的问题：**把被 compaction 和 session 边界撕裂的讨论碎片自动打捞、提炼、关联，让 Agent 可以跨时间线看到完整的主题脉络。**

---

## 二、这个事实带来的问题

因为会话之间不共享记忆，用户会反复遇到重复解释、决策被推翻、TODO 丢失、上下文断裂等问题——新 session 的 Agent 不知道上一个 session 讨论过什么、决定了什么。这不是模型能力问题，而是 architecture 问题。

---

## 三、现有机制为什么不够

Reasonix 已经有三种与记忆相关的能力，但它们各自有明确的边界：

| 机制 | 作用 | 为什么不够 |
|---|---|---|
| **会话内上下文** | 理解当前对话 | 容量有限，会话结束就清空 |
| **显式记忆 (memory/)** | 用户/Agent 主动保存的结论 | 扁平目录，无索引，不保存决策过程 |
| **会话日志 (archive/)** | 完整记录每次对话 | 只存不索引，无法检索，无法直接消费 |

显式记忆能保存结论，但不能回答"这个结论是怎么来的"。会话日志里有过程，但没有任何结构，Agent 无法直接消费。

所以，我们需要一种新机制：**自动从会话日志里提取有价值的信息，并让这些信息跨会话可用。**

---

## 四、核心目标：主题追溯

我们要解决的根本问题是：

> **同一个话题的讨论和决策，散落在多个 session 里，Agent 无法跨时间线把它们打捞出来、串成一条完整的演进线。**

例如"auth-flow"这个话题：

- 第 X 天你决定用 JWT + httpOnly cookie；
- 数日后你实现了登录接口；
- 一周后你为 Safari 兼容性调整了 cookie 策略；
- 数周后新 session 启动，Agent 却建议把 refresh token 放 localStorage。

如果 Agent 能看到 `auth-flow` 这个主题的完整时间线，它就不会提出与早期决策相矛盾的建议。它会问：

> "我们周一明确排除了 localStorage。周五的调整是为了解决 Safari 问题。你这次修改是想推翻原决策，还是只是补充？"

这种能力我们称为 **主题追溯（Theme Tracing）**。

```text
时间轴
──────────────────────────────────────────────────────►
  第X天          数日后          一周后           数周后
   │               │                │                │
   ▼               ▼                ▼                ▼
[JWT 方案]    [登录接口实现]    [Safari 调整]    [Agent 建议 localStorage]
   │                                                   │
   └────────────────── auth-flow 主题 ──────────────────┘
```

## 五、从目标倒推：我们需要哪些能力？

主题追溯——让 Agent 跨 session 打捞相关讨论、串联决策时间线——听起来让人心动，但也容易让人觉得玄幻：这真的不是一厢情愿的幻想吗？

答案是不玄幻。原因是一个简单的事实：

> **你在长周期项目上发生的所有讨论、决策、争论、折衷，都已经通过对话轮历史，持久化地保存在你的硬盘里。**

它们就在 `archive/` 下的 JSONL 文件中。每次回车、每次工具调用，都被完整记录。

所以问题不是"这些信息是否存在"，而是"如何把它们打捞出来"。只要建立合理的打捞 → 提炼 → 主题归纳机制，让 Agent 在需要时能找到相关历史，主题追溯就是可行的。

接下来我们一步步拆解这个机制需要什么。但先看一个整体流程图，了解搜索→簇→提炼→视图→主题的完整链路：

<img width="800" height="460" alt="search-flow" src="https://github.com/user-attachments/assets/17f468d8-1278-43d0-a822-db88d741105c" />


核心概念：
- **Cluster（簇）**：搜索命中一个关键词后，不仅返回命中 turn，还会按时间窗口（约 90 秒 gap）向左右扩展相邻 turns。多次连续交互大概率包含一次完整的决策/事件，这就是一个 cluster。
  *例子：搜索"cookie"命中一个 turn，系统自动带上此前后 90 秒内的 turns——前一条还在讨论"JWT 方案"，后一条已经开始确认"那就定 cookie 了"。这三条 turns 一起组成一个 cluster，描述了一次完整的决策过程。*
- **Search View（搜索视图）**：每次搜索产生的 clusters 结构被保存为一个 JSON 文件。后续 LLM 可以通过查看搜索视图来了解"上次搜 auth 找到了哪些候选材料"，不必重复搜索。
- **多次搜索 → 多视图 → 主题**：追溯一个话题可能需要从不同角度（"JWT"、"cookie"、"登录"、"Safari"）多次搜索。每次形成独立的 search view。LLM 通过这些视图的 keyword 串联，寻找最相关的 turns，按相关性挂载到 theme。

### 5.1 原始日志无法直接打捞

会话日志里虽然有信息，但它太长、太吵、没有结构，Agent 没法直接读。

所以我们需要一种**把原始日志压缩成可检索材料**的能力。这种能力称为 **提炼（Refine）**。

### 5.2 需要多角度搜索历史材料

即使有了提炼后的摘要，也需要一种方式让用户或 Agent 说"我要找 auth 相关的内容"，然后返回所有 session 里的相关片段。

这种能力就是 **跨会话搜索**。

搜索不是简单返回命中的第一条 turn。实际机制是（来自 MCP 源码 `context-tools.ts`）：

1. **关键词命中**：在会话日志中匹配关键词，找出所有命中 turns
2. **聚簇（Cluster）**：相邻命中 turns 被分组为 blocks，每个 block 从命中位置左右扩展（约 90 秒 gap），带回相邻 turns 形成 cluster。多次连续交互大概率包含一次完整决策过程的单元
3. **提炼**：cluster 中未被提炼的 turns 被批量处理，结果存入 SQLite `refined_turns` 表
4. **保存搜索视图**：每次搜索的结果——包括 query、所有 clusters、提炼统计——被保存为一个 JSON 文件

搜索视图的结构（实际源码 `context-tools.ts:375-390`）：

```json
{
  "query": "auth JWT cookie",
  "createdAt": "2026-06-25T10:00:00Z",
  "totalMatches": 5,
  "totalRefined": 12,
  "clusters": [
    {
      "sessionId": "session-A",
      "hitTurnId": 12,
      "memberCount": 3,
      "members": [
        { "sessionId": "session-A", "turnId": 10, "timestamp": "..." },
        { "sessionId": "session-A", "turnId": 11, "timestamp": "..." },
        { "sessionId": "session-A", "turnId": 12, "timestamp": "..." }
      ]
    }
  ]
}
```

### 5.3 搜索过程本身也是打捞过程

搜索过程中，命中的 turns 及其 cluster 成员被自动提炼入库。每次搜索产生一个 search view，作为主题挂载的候选材料集。

追溯一个话题可能需要从不同角度多次搜索——例如"JWT"、"cookie"、"Safari"——每次产生一个独立的 search view。

### 5.4 分析搜索视图并挂载主题

多个搜索视图构成候选材料库。LLM 通过 keyword 串联这些视图，分析每个命中的 cluster 的内容，判断哪些 genuinely 属于同一个主题，然后挂载到 theme 下。

挂载之后，theme 就形成了一条按时间线排列的历史档案，包含该主题的所有决策历史和修改历史。LLM 在挂载过程中已经深度分析了所有相关 turns，因此对主题的理解甚至比人类用户更清晰（见"挂载即理解"）。

这四个能力的依赖关系是：

```text
原始会话日志
    ↓ 无法直接打捞
   提炼
    ↓ 生成可检索材料
   搜索 + 搜索视图
    ↓ 命中即提炼，扩展材料库
   材料库（Refined + Search Views）
    ↓ Agent/LLM 分析相关性
   主题挂载
    ↓
   主题时间线
```

## 六、提炼：把原始日志变成可打捞的材料

### 6.1 为什么需要提炼

§五 已经解释了原始日志不可直接用于主题追溯——体积大、噪音多、无结构、无索引。提炼是解决这个问题的第一步：把原始日志转换成一种**可被检索、可被分析**的形态。

### 6.2 提炼做什么

提炼的目标是从一次 turn 中提取出对主题追溯有用的信息：

- 用户这次想做什么；
- Agent 做了什么决定或建议；
- 涉及哪些文件；
- 调用了哪些工具；
- 是否出现错误；
- 接下来计划做什么。

提炼后的产物称为 **Refined Turn**，是对原始 turn 的压缩和结构化。它保留了检索价值和分析价值，去掉了无关的闲聊和中间试探。

### 6.3 为什么提炼必须是自动的

如果让用户或 Agent 在每轮对话后手动写摘要，提炼质量会很高，但负担太大。用户不可能每次交互后都做这种判断。

所以提炼必须是自动的。

### 6.4 为什么不用 LLM 做提炼

用 LLM 做提炼看似自然，但会带来几个问题：

| 问题 | 说明 |
|---|---|
| **成本** | 每段对话都要消耗 token，与 Reasonix 追求的低成本相矛盾 |
| **延迟** | 需要等待模型返回，影响实时体验 |
| **不可复现** | 相同输入可能产生不同输出，难以调试 |
| **不可解释** | 不知道系统为什么把某段话标记为决策 |
| **错误会污染下游** | 一旦提炼错了，搜索和主题追溯都会被误导 |

### 6.5 确定性规则提炼

因此，我们选择**基于关键词和规则的确定性提炼**。它的工作方式是：

- 识别动作信号词，例如"决定"、"完成"、"修复"、"下一步"；
- 识别 Markdown 标题，例如"当前任务"、"阻塞项"、"决策"；
- 从工具调用参数中读取涉及文件；
- 从工具返回中读取错误信息。

> **注意 Reasonix JSONL 特性**：`role: assistant` 的 entry 包含 `reasoning_content`（推理过程文本）和预调用的 `tool_calls`。提炼时应**只处理 `content` 字段**，忽略 `reasoning_content`——推理文本噪音大、长度长，不适合作为搜索信号。`tool_calls` 中的预调用也不重复计入，以 `role: tool` 实际结果为准。

这种方式不完美，但足够便宜、快速、可解释。它的设计目标不是 100% 召回，而是：

> **提供一个"够用的材料库"，让搜索能找到大部分相关内容，再用主题挂载手动补充遗漏。**

一个提炼后的 turn 大概长这样：

```json
{
  "sessionId": "session-A",
  "turnId": 12,
  "summary": "确定 Auth 使用 JWT + httpOnly cookie",
  "facts": ["Auth 使用 JWT + httpOnly cookie"],
  "entities": {
    "files": ["src/auth.rs"],
    "tools": ["read_file", "edit_file"],
    "errors": []
  },
  "categories": {
    "decisions": ["Auth 使用 JWT + httpOnly cookie"]
  }
}
```

### 6.6 多角度搜索

提炼之后，系统就有了一个可检索的材料库。用户或 Agent 可以通过关键词搜索，把和某个话题相关的 turns 捞出来。

同一个主题可能用不同关键词描述。多角度搜索会产生多个临时结果集，共同构成更完整的材料库。

### 6.7 搜索视图

为什么是搜索触发提炼，而不是全量提炼？因为**搜索本身就是一种隐式权重**。

如果某个话题从未被搜索过，说明它没人关心——那么它的 turns 留在原始 JSONL 中就好，不需要占用提炼空间和索引资源。只有当某人（用户或 Agent）主动搜索"auth"时，才说明这个话题当前有价值，才值得被提炼、被挂载、被追溯。

全量提炼看起来更"完整"，但实际上会引入大量噪音——每次 session 中的日常调试、过程确认、临时命令——这些 turns 在没人搜索的时候只是占用空间的字节。搜索触发提炼天然做了第一轮信号过滤：**搜索即权重，无人提及的决策就让它安静地留在原始记录中。**

每次搜索会产生一个 **搜索视图（Search View）**。它不是最终结果，而是一个临时材料集合，供进一步分析使用。

搜索视图的价值在于：

- 让 Agent 可以回顾"我上次搜 auth 时找到了哪些材料"；
- 让 Agent 不必重复搜索；
- 作为主题挂载的候选来源。

一个搜索视图的结构如下（对应源码 `context-tools.ts` 中的实际格式）：

```json
{
  "query": "auth JWT cookie",
  "createdAt": "2026-06-25T10:00:00Z",
  "totalMatches": 5,
  "totalRefined": 12,
  "clusters": [
    {
      "sessionId": "session-A",
      "hitTurnId": 12,
      "memberCount": 3,
      "members": [
        { "sessionId": "session-A", "turnId": 10, "timestamp": "..." },
        { "sessionId": "session-A", "turnId": 11, "timestamp": "..." },
        { "sessionId": "session-A", "turnId": 12, "timestamp": "..." }
      ]
    }
  ]
}
```

### 6.8 Cluster：搜索的基本单位

搜索命中一个关键词后，不会只返回命中 turn。系统会按时间窗口（约 90 秒——实测的经验值，可配置）向左右扩展相邻 turns，形成一个 **cluster**（簇）。连续多次交互大概率包含一次完整的决策或事件。

```
时间轴 →→→→→→→→→→→→→→→→→→→→→→→→→
  turn-10  turn-11  turn-12(HIT)  turn-13  turn-14
  ──────────┼───────────┼──────────┼───────────
  ← 左扩展(90s) →      命中      ← 右扩展(90s) →
  └─────────────── cluster ────────────────┘
```

Cluster 是搜索和提炼的基本操作单位。它在搜索时自动形成，被提炼后以 `refined_turns` 存入 SQLite，同时被记录在搜索视图的 `clusters[]` 中。

> **参数说明**：时间窗口默认为 90 秒（实测经验值，可通过 `cluster_gap_seconds` 调整）。相邻命中 turns 会先合并为 **block** 再整体扩展，避免每个命中单独成簇。单个 cluster 的大小受 `max_cluster_size` 限制（默认 15，建议调至 5-8），防止过度膨胀。

> **注意**：cluster 机制是搜索质量的优化，不是必要条件。即便不依赖 cluster——只靠 `search_context` 关键词命中 → LLM 分析相关性 → `tag_theme` 挂载——也能建立基本的 theme 实现跨 session 追溯。Cluster 的作用是提高打捞效率：命中关键词后自动带回相邻上下文，让 LLM 在挂载时有更完整的材料可分析。如果 cluster 的时间窗口参数不完美，影响的是效率而非可行性。

> **未来演进**：当前 cluster 基于固定时间窗口，假设同窗口内的连续 turns 包含一次完整决策。这个假设在多数场景下成立，但不完美。如果方案被验证有价值，可以考虑引入轻量语义分析库，对归档 turns 做**语义聚类**：分析 turn 之间的语义关联，自动发现 cluster 边界，甚至可以进一步**归桶（Bucket）**——将相似 cluster 自动分组为主题候选材料。归桶可以看作是主题的预加工，为后续 LLM 挂载提供更高质量的材料。这一能力可以放在 Phase 2 或 Phase 3 实现，作为关键词聚簇的可选升级路径。

### 6.9 搜索即打捞

搜索过程中，每个 cluster 中的未提炼 turns 会被自动提炼并入库。这意味着：

> **搜索不只是查询，也是打捞。**

你搜索"auth"的时候，系统不仅返回已有的 refined turns，还会把历史 session 里和"auth"相关的未提炼 turns 一起处理，变成可用材料。

### 6.10 材料库的构成

到这里，我们可以明确材料库的构成：

```text
材料库 = Refined turns（已提炼的结构化摘要）
       + Search Views（搜索产生的临时候选集）
```

用图表示：

```text
┌─────────────────────┐      ┌─────────────────────┐
│   Refined turns     │      │    Search Views     │
│  (SQLite 材料库)    │      │  (JSON 临时候选集)  │
│                     │      │                     │
│  session-A/12       │      │  query: auth        │
│  session-B/3        │      │  clusters: [...]    │
│  session-C/27       │      │                     │
└──────────┬──────────┘      └──────────┬──────────┘
           │                            │
           └────────────┬───────────────┘
                        ▼
              ┌─────────────────────┐
              │     材料库          │
              │  Agent/LLM 分析基础 │
              └─────────────────────┘
```

材料库是 Agent/LLM 分析并挂载主题的基础。

## 七、主题：把材料组织成跨时间的追踪线

### 8.1 为什么是主题，而不是标签或文件夹？

现有显式记忆通常用文件夹分类，例如 `memory/decisions/`、`memory/rules/`。Reasonix 的 `memory/` 就是这种模式：文件存放在 `memory/global/` 下，通过 `remember` 工具写入，通过文件名关键词搜索。文件夹适合静态分类，但不适合追踪一个议题在时间上的演变。

标签比文件夹灵活，但标签只回答"这段内容包含什么关键词"，不回答"这段内容属于哪个长期议题"。

我们需要的是一种**跨时间的追踪线**，能够把不同 session 里关于同一议题的 turns 串起来，形成时间线。这种组织方式称为 **Theme（主题）**。

### 8.2 主题的特征

一个主题的特征：

- 它是**语义组**，不是关键词标签；
- 它**跨 session**，可能覆盖数周甚至数月；
- 它**可演进**，能记录议题从提出、讨论、决策到修改的全过程；
- 它只存**引用**，不复制原文。

例如 `auth-flow` 这个主题可以包含：

- 第 X 天决定用 JWT；
- 数日后实现登录接口；
- 一周后为 Safari 兼容性调整 cookie 策略；
- 数周后 review 时的讨论。

### 8.3 谁决定挂载主题

材料库里的候选片段需要判断哪些 genuinely 属于同一个主题。这个判断由 Agent/LLM 完成，然后挂载。

系统不推荐自动挂载，因为：

- 自动挂载会导致主题膨胀、噪音增加；
- 只有理解上下文的 Agent/LLM 才能判断"这段话 genuinely 属于 auth-flow 还是只是提到了 auth"。

### 8.4 主题时间线

挂载之后，通过 `trace_theme` 可以按时间顺序查看主题的完整演进：

- 这个议题第一次是什么时候讨论的？
- 中间做过哪些决策？
- 哪些文件被反复修改？
- 最近一次结论是什么？

主题时间线是主题追溯的最终产物，也是后续决策关联分析的基础。

一个主题文件只保存引用，不复制原文：

```json
{
  "theme": "auth-flow",
  "displayName": "Auth 认证流程决策演进",
  "turns": [
    { "sessionId": "session-A", "turnId": 12, "timestamp": "2026-06-20T10:30:00Z" },
    { "sessionId": "session-A", "turnId": 14, "timestamp": "2026-06-20T10:35:00Z" },
    { "sessionId": "session-B", "turnId": 3,  "timestamp": "2026-06-22T09:00:00Z" }
  ],
  "memories": [
    { "key": "auth-jwt-cookie", "folder": "memory/decisions", "timestamp": "2026-06-20T10:40:00Z" }
  ]
}
```

```text
session-A          session-B          session-C
   │                  │                  │
   ▼                  ▼                  ▼
 [turn 12]         [turn 3]           [turn 27]
   │                  │                  │
   └──────────────────┴──────────────────┘
              │
              ▼
        auth-flow 主题
              │
              ▼
        跨 session 时间线
```

---

### 挂载即理解

主题追溯中有一个看似无关的步骤：把搜索打捞到的 turns 逐个判断"这个 turn 真的属于 auth-flow 吗"。

这个判断由 LLM 完成。但它的价值不止于归类正确——**LLM 分析每个 turn、判断它是否属于某个主题的过程，本身就是让 LLM 深入理解该主题的过程**。

LLM 看了 turn-12 "JWT 方案讨论"，turn-14 "cookie 存储方案确认"，turn-3 "Safari 兼容性调整"——每段都经过语义分析。当挂载完成时，它已经对整个主题的发展脉络有了**比人类用户更清晰的事实基础**。因为用户可能只参与了其中 2 个 session，而 LLM 看了全部 8 个相关 session。

这就是为什么 LLM 来做这件事，不是成本问题，而是**战略选择**。每次主题挂载既是一次归类操作，也是一次主题理解训练。随着时间的推移，LLM 对项目历史决策的理解会越来越深——它不是被强制才去检查历史，而是自己就"记得"。

### 主题的调用闭环

`trace_theme` 的触发有三种情景：

1. **用户主动触发**——用户直接请求"帮我查一下 auth 相关的历史讨论"
2. **Agent 关键任务前自动调用**——当 Agent 准备执行较大的或关键的任务时，通过 Prompt 指引它先 `trace_theme` 了解历史决策
3. **TodoList 第零项**——当 Agent 生成了较长的 todo list（如 5 项以上）时，系统自动在第零项插入一个可选的 `trace_theme` 任务：*"🔍 检查与本次改动相关的历史主题（如 auth-flow）"*。Agent 可以选择执行或跳过——这是一个强提醒，但不强制。

三种情景叠加，保证 Agent 在关键节点有机会想起检查历史，但不每次都打断它。

更重要的是**主题维护闭环**：当 Agent 主动发起 trace 并找到当前任务可挂载的主题时，Prompt 应同时提示它将当前相关的 turns 提炼后挂载到该主题。这样每个关键任务的执行，同时也是主题的**创建→调用→维护**闭环——主题不会因为一次创建就过期，而是随着项目推进持续生长。

```
          用户主动请求 / 关键任务触发
              │
              ▼
          trace_theme
              │
     ┌────────┴────────┐
     ▼                 ▼
  命中主题         未命中
     │                 │
     ▼                 ▼
  挂载当前 turns   创建新主题
  到已有主题           │
     │                 ▼
     └───────┬─────────┘
             ▼
       主题持续生长
```

> **决策守卫**：上述调用闭环是主题追溯的自然使用方式。在它之上还有一个更高层的概念——决策守卫。它通过 Prompt 引导 Agent 在每次执行决策敏感操作前，主动搜索记忆并 `trace_theme`，确认本次决策与历史的一致性。决策守卫不拦截、不强制，而是让检查历史决策成为 Agent 最自然的行为选择。详细设计见另一篇文章。

## 八、三层存储架构

把上面的设计选择串起来，就得到了 Reasonix 记忆系统的三层存储架构：

<img width="900" height="520" alt="contextFlow" src="https://github.com/user-attachments/assets/38e7ce46-6f8f-44f8-9b94-68f2e1f63f63" />


*四段会话（sessionA~D）按时间线排列。每个小色块是一个 turn（对话轮），色块聚类的线段是 cluster（决策/事件单元），跨 session 的括号是 theme（主题线索）。*

```text
┌─────────────────────────────────────────────┐
│  第一层：Raw（原始日志）                       │
│  archive/YYYYMMDD-HHMMSS.jsonl                │
│  只读审计，不被 Agent 直接消费                │
├─────────────────────────────────────────────┤
│  第二层：材料库（Material Library）            │
│  .reasonix/refined/refined.sqlite            │
│  .reasonix/searches/*.json                   │
│  可检索、可分析的历史材料集合                 │
├─────────────────────────────────────────────┤
│  第三层：主题关联（Thematic）                  │
│  .reasonix/themes/*.json                     │
│  跨 session 的相关 turns / memories 引用集合 │
├─────────────────────────────────────────────┤
│  索引与精要                                  │
│  .reasonix/index.json                    │
└─────────────────────────────────────────────┘
```

### 10.1 Raw：原始日志

Reasonix 已经会把每次会话写成 JSONL。我们**不改这个格式**，只是读取它。原始日志是只读审计层，Agent 不直接消费它。

> **关于 turn ID**：当前 JSONL 没有显式的条目标识，refined_turns 通过 `(sessionId, turnIndex)` 推算行号指向原文。如果社区给每条 JSONL entry 加上 `turnId` 字段（详见附录），`refined_turns.turn_id` 可以直接定位原始 entry，溯源链路更稳健。

### 10.2 材料库

材料库由两部分组成：

- **Refined**：自动提炼的结构化摘要，存储在 SQLite 中；
- **Search Views**：搜索产生的临时候选集，存储在 JSON 文件中。

两者共同支撑主题挂载和主题时间线。

### 10.3 Thematic：主题关联

Theme 是跨 session 的议题追踪线。它只保存引用，内容仍存放在材料库和原始日志中。这样即使主题文件被删除，原始数据也不会丢失。

### 10.4 索引与精要

- **index.json**：记录 memory、notes 的元数据，用于快速列出和搜索；
- **essence.md**：由 Agent 定期整理生成的项目精要，启动时加载。

## 九、为什么不用向量/图数据库？

向量数据库 + 大规模 RAG 的思路，本质上是"把所有历史对话都存下来，用的时候靠语义相似度捞一捞"。这对 Reasonix 有问题：

- 记忆失去筛选机制；
- "语义相似"在 coding 场景中经常返回干扰项；
- 每条对话都要生成 embedding，成本高且不可解释。

图数据库、时序知识图适合复杂关系建模，但对终端 coding agent 来说：

- 需要额外服务和运维；
- 每条对话都要实体抽取、关系链接，写放大严重；
- 开发者不会为了让 Agent 记住而先去维护知识图谱。

所以我们选择本地 SQLite + Markdown：SQLite 保存自动提炼的索引，Markdown 保存用户主动写入的显式记忆。

## 十、最小实现：用 MCP 插件验证

### 12.1 目标

最小实现的目标是：用最小的成本，验证主题追溯是否有价值。

它不需要修改 Reasonix 核心代码，只通过一个 MCP 插件接入。

### 12.2 最小能力集合

一个能跑通主题追溯闭环的 MCP 插件，需要具备以下能力：

| 能力 | 作用 |
|---|---|
| **适配器** | 把 Reasonix 的 JSONL 格式转换成统一的 turn 结构 |
| **提炼** | 把 turns 压缩成可检索的结构化摘要 |
| **搜索** | 基于关键词跨 session 打捞相关材料 |
| **搜索视图** | 保存每次搜索的临时结果 |
| **主题挂载/追溯** | 把相关 turns 挂到主题，并按时间线展示 |
| **启动加载** | 每次运行开始时把历史上下文推送给 Agent |

**Phase 1 验证标准**（用户用来判断"跑通了"）：

1. 对任意 session 执行 `search_context("关键词")`，能返回该 session 中相关的 turns
2. 搜索同一关键词在多个 session 中都返回结果
3. 每个结果包含：摘要、文件引用、该 turn 的类别（decision / bugfix / arch 等）
4. 如果关键词比较少见（如特定文件名），搜索结果第一页就能命中目标

这四条全部满足，说明提炼质量够用、搜索路径正确，可以进入 Phase 2。

```text
Reasonix
    │
    │ 写入
    ▼
archive/YYYYMMDD-HHMMSS.jsonl
    │
    │ 适配器切分 turns
    ▼
提炼引擎 ──► refined.sqlite
    │
    │ search_context
    ▼
搜索视图 ──► Agent/LLM 分析
    │
    ▼
tag_theme ──► themes/*.json
    │
    ▼
trace_theme ──► 主题时间线
```

### 12.3 验证标准

在考虑进入下一阶段之前，先回答三个问题：

1. 提炼质量是否够用？用户搜索 auth 时，能否找到相关的决策 turns？
2. 主题挂载是否有效？Agent 是否能从搜索材料中准确判断哪些属于同一主题？
3. 用户是否感知到价值？Agent 在推荐方案前，是否会主动搜索历史并引用主题时间线？

如果三个问题都是肯定的，再进入原生集成阶段。

## 十一、中期进化：原生集成

### 13.1 为什么原生集成

MCP 模式适合验证，但原生集成有明显优势：

| 场景 | MCP 插件 | 原生内建 |
|---|---|---|
| 提炼触发时机 | 搜索命中时自动触发 | session 结束或 compaction 时即时触发 |
| 启动加载 | 需 Agent 主动调用 | 可在启动流程中自动注入 |
| 错误处理 | MCP 连接中断后需重试 | 主进程内，无连接问题 |
| 与 todo list 集成 | 无法感知 LLM 输出 | 可在 planning 层注入 |

### 13.2 内建范围

原生集成后，以下能力成为 Reasonix 内建工具：

- `search_context`：跨 session 搜索；
- `tag_theme`：挂载主题；
- `trace_theme`：查看主题时间线；
- `refine_all`：提炼所有未提炼 session。

同时，在以下生命周期点自动触发提炼：

- session 结束时；
- compaction 时；
- 搜索命中时。

> **注意**：即使原生集成，也不推荐在无精确语义分析的情况下做全量提炼。全量提炼会将大量日常过程性对话（调试、确认、临时命令）也纳入索引，稀释搜索信噪比。搜索触发提炼的"搜索即权重"原则依然成立。只有当引入语义分析库可做精确的提簇和归桶时，才考虑全量预处理，实现搜索驱动和语义驱动**双向归并**的冗余策略。

搜索同时也是打捞过程。`search_context` 命中未提炼的 turns 时自动触发提炼入库。搜索不只查询，也是拓展材料库。

## 十二、长期愿景

### 14.1 记忆像操作系统文件系统

Reasonix 的长期记忆能力应该像操作系统的文件系统一样：**无处不在、默认启用、不打扰用户，但在需要时随时可用**。

### 14.2 用户可控

用户必须始终能控制：

- 是否启用自动提炼；
- 是否启用历史决策检查；
- 查看和编辑提炼结果；
- 删除历史；
- 导出数据。

### 14.3 扩展生态

即使核心功能原生化了，MCP 仍然可以作为策略扩展口存在：

- 第三方可以用 LLM 实现更智能的提炼策略；
- 超大规模项目可以接入向量检索后端；
- 企业可以接入共享记忆服务；
- 特定领域可以有专门的提炼器。

最终形态：**原生核心 + MCP 扩展口**。

### 14.4 性能预期

> 实际测量：一个中等规模的 Reasonix 会话（~2000 行 JSONL）含 ~180 个用户 triggered 的交互单元，提炼后约 200KB。

| 规模 | 每天数据 | 年数据 | 说明 |
|---|---|---|---|
| 轻度用户：5 sessions × 80 turns/day | 400KB | 146MB | SQLite 零压力 |
| 重度用户：20 sessions × 150 turns/day | 3MB | 1.1GB | 建议按年归档旧 theme |
| 团队项目：100 sessions × 150 turns/day | 15MB | 5.5GB | 建议定期清理过时主题 |

## 十三、产品边界与体验原则

### 15.1 系统做什么

1. 自动提炼会话中的关键信息；
2. 提供跨会话关键词搜索；
3. 保存搜索视图；
4. 组装主题时间线；
5. 在合适时机提醒 Agent 检查历史决策。

### 15.2 Agent 做什么

1. 判断搜索命中的材料是否 genuinely 属于某个主题；
2. 选择挂载主题；
3. 解释主题时间线；
4. 最终决定是否采纳历史决策。

### 15.3 用户做什么

1. 查看和编辑主题文件；
2. 启用或关闭相关能力；
3. 纠正错误的主题挂载。

### 15.4 体验原则

- **不要替用户做判断**：系统可以推荐，但主题挂载由 Agent/用户决定；
- **不要打断正常流程**：主题追溯是后台能力；
- **保持可审计**：主题文件是透明的，用户可以随时查看；
- **渐进生效**：主题越多、越准确，价值越大。

---

## 十四、验证路径

整个设计建议按以下顺序验证：

1. **Phase 1**：用 MCP 插件验证"提炼 + 跨会话搜索 + 主题挂载/追溯"是否有用户价值；
2. **Phase 2**：如果价值被验证，把核心能力提升为 Reasonix 内建工具；
3. **Phase 3**：当主题库足够丰富后，进入长期维护与策略扩展阶段。

每一步都可以独立回滚，不影响现有功能。

## 十五、总结

Reasonix 记忆系统的设计逻辑可以概括为一条因果链：

```text
会话独立 → 历史决策散落
    ↓
需要主题追溯
    ↓
需要打捞散落在各 session 中的 turns
    ↓
原始日志无法直接打捞 → 引入提炼
    ↓
提炼形成可检索材料 → 引入搜索与搜索视图
    ↓
材料库 = Refined + Search Views
    ↓
Agent/LLM 分析材料 → 挂载主题
    ↓
主题形成时间线 → 后续决策关联分析
```

核心原则是：

- **主题追溯是核心目标**；
- **提炼是打捞手段**；
- **搜索视图和 refined turns 共同构成材料库**；
- **主题是跨时间的追踪线**；
- **主题关联分析是可选的下游能力**；
- **本地轻量、用户可控、渐进验证**。

这不是一个需要重写 Reasonix 的宏大计划。它是一个可以从 MCP 插件开始、逐步原生化的渐进式能力。但如果它工作得好，Reasonix 用户将不再在每个新会话里“重新认识”自己的项目。

## 附录：社区集成建议

如果社区采纳了本方案的价值，以下是从 MCP 过渡到原生集成的考虑：

| 场景 | MCP 插件 | 原生内建 |
|------|---------|---------|
| compaction 时自动 refine | 需轮询 JSONL 目录 | 零成本事件触发 |
| 决策守卫（TodoList 第零项） | ⚠️ 无法感知 LLM 输出 | Prompt 引导 Agent 在关键任务前自动 trace_theme + 搜索记忆，或通过 TodoList 第零项强提醒 |
| 提炼触发时机 | 搜索命中时自动触发 | 会话结束时即时触发 |

原生集成的最小侵入范围：新增 `refine/` 子目录（与 `memory/` 同级）、新增 4 个内建工具（`refine_session`、`search_context`、`tag_theme`、`trace_theme`）、在 compaction 流程中插入一次 refine 调用。不改任何现有数据结构。

### 给 JSONL 加上 turn ID

当前 Reasonix 的 JSONL 按行顺序记录对话，没有显式的 turn 编号。这意味着 refined turns 无法直接引用原始的 JSONL entry——需要按 `(sessionId, turnIndex)` 推算行号，不精确也不高效。

建议社区考虑给每条 JSONL entry 加上 `turnId` 字段：

```jsonl
{"turnId": 1, "role": "user", "content": "帮我看看 auth 模块"}
{"turnId": 2, "role": "assistant", "content": "好的。", "reasoning_content": "...", "tool_calls": [...]}
{"turnId": 3, "role": "tool", "content": "{file content}", "name": "read_file", "tool_call_id": "call-1"}
```

这样 SQLite 中的 `refined_turns.turn_id` 可以直接指向原始 JSONL 的 entry。当 LLM 需要还原某个主题上某一天某条决策的对话原文和操作详情时，可以按图索骥——`theme → cluster → refined_turn → turnId → 原始 JSONL → 完整上下文`。

即使先不加 `turnId`，本设计也能工作（通过 `(session_index, turn_index)` 推算行号），但加上了会更稳健、更高效。这是一个低成本的社区改进建议。

即使原生集成后，MCP 模式仍应保留作为扩展口——第三方可以用自己的提炼策略（语义分析、向量搜索等）通过 MCP 注册替代默认实现。最终形态：**原生核心 + MCP 扩展口**。

### 参考实现

本设计已有一份完整的 MCP 服务器参考实现，基于 Kimi Code 构建，开源在 [github.com/Zehee/kimi-code-memory-mcp-server](https://github.com/Zehee/kimi-code-memory-mcp-server)：

- `search_context`——关键词命中 + 左右扩展聚簇 + 自动提炼
- `tag_theme` / `trace_theme`——主题挂载与时间线追溯
- `list_search_views` / `list_themes`——搜索视图和主题列表管理
- SQLite 存储 refined turns，JSON 存储 search views 和 themes
- 与 Reasonix 的差异仅在于 JSONL 格式（Kimi Code 的 `wire.jsonl` vs Reasonix 的 `archive/JSONL`），适配层约 40 行

该实现可以作为 Phase 1 的起点，只需要替换 JSONL 适配器即可接入 Reasonix。
